# OpenScanLink - 开发总结与踩坑记录

本文档总结了本项目从前期重构到底层扫描引擎替换，再到大规模功能完善、Bug 修复及打包发布的完整开发过程，特别是开发过程中遇到的各类“坑”及对应的解决方案。

## 1. 项目概览与技术栈
- **项目名称**: OpenScanLink
- **前端架构**: Vue 3 + Vite + Element Plus + Pinia
- **桌面端架构**: Electron (主进程与渲染进程 IPC 通信)
- **底层扫描引擎**: C# (.NET 8) + `NAPS2.Sdk` (替代了原有的 WIA) + `Magick.NET` (图像后处理)
- **通信机制**: 网页端通过 HTTP (默认 `http://127.0.0.1:17890`) 调用本地 Electron 服务进行扫描控制。

## 2. 核心功能实现总结
1. **底层引擎重构**: 成功将扫描核心从 WIA 迁移至 `NAPS2.Sdk`，实现了**连续高速扫描**和**双面扫描**功能，摒弃了原生的 Windows 扫描弹窗，实现静默扫描。
2. **图像后处理**: 接入 `Magick.NET`，实现了去底灰、黑白二值化 (`WhiteThreshold` / `BlackThreshold`)、自动纠偏 (`Deskew`) 和边缘裁剪去黑边 (`Trim`)。
3. **通信与状态同步**: Node.js 主进程通过 IPC (`scan-task-update`) 实时向 Vue 渲染进程同步多页扫描的任务进度。
4. **系统级交互**: 实现了网页端连接成功后的 Windows 右下角系统弹窗提示；增加了本地图片缓存的“一键清空”和“打开文件夹”功能。
5. **版本更新机制重构**: 弃用了受限的 `electron-updater` (`latest.yml`)，改用基于远程服务器自定义 JSON 配置的更新机制。
6. **配套文档与 Demo**: 编写了供第三方业务系统对接的《网页对接文档.md》，并在 `web/demo` 下提供了一套完整的 Vue3 测试页面。

## 3. 核心踩坑记录与解决方案 (Pitfalls & Solutions)

### 🔴 坑 1: TWAIN 驱动崩溃与“参数错误”
- **现象**: 扫描仪刚启动就崩溃，或者第二页扫描到三分之一卡住，报错。
- **原因**: C# 底层误删了 `SetUpWin32Worker()`，导致 TWAIN 驱动对象被 .NET 垃圾回收器 (GC) 强制回收。
- **解决**: 恢复了 `scanningContext.SetUpWin32Worker()`，并在 `.csproj` 编译配置中确保 `NAPS2.Worker.exe` 被正确拷贝至 `bridge` 运行目录下，使其作为独立进程挂载。

### 🔴 坑 2: 生产环境打包后找不到扫描仪 (Mock 设备)
- **现象**: `npm run dev` 运行完美，但 `npm run build:win` 打包成 exe 安装后，设备列表显示“未找到驱动 (Mock 设备)”。
- **原因**: Electron 默认将所有代码打包进只读的 `app.asar` 黑盒中，`child_process.execFile` 无法执行 `.asar` 内部的 `ScannerService.exe`。
- **解决**: 在 `package.json` 的 `build` 配置中，将 `bridge` 文件夹放入 `extraResources`，并在 `index.ts` 中通过 `app.isPackaged ? process.resourcesPath : process.cwd()` 动态解析物理绝对路径。

### 🔴 坑 3: 安装后体积暴增 (高达 1.14GB)
- **现象**: 仅一个简单的扫描工具，打包出来的 `.exe` 安装包安装后，安装目录体积超过 1GB。
- **原因**: 
  1. `extraResources` 把 C# 的整个源码目录、编译缓存 (`obj`, `bin`) 以及上百 MB 的 `.pdb` 调试文件全拷进去了。
  2. `package.json` 中误将 `vue`, `vite`, `typescript` 等纯前端构建工具放到了 `dependencies` 而非 `devDependencies`，导致 Node 模块被全量打包。
- **解决**: 严格配置 `extraResources` 的 `filter`（排除 `!ScannerService/**/*`, `!*.pdb`）；整理 `package.json` 依赖，将安装后体积极限压缩至约 **534MB**。

### 🔴 坑 4: NSIS 安装路径无法自动补全自定义文件夹
- **现象**: 安装时，点击浏览选择 `D:\` 盘，路径直接变成了 `D:\`，而不会像其他软件那样自动加上 `D:\OpenScanLink`。
- **原因**: `electron-builder` 底层处理包含特殊字符的 `productName` 时变量解析丢失，导致追加路径为空。
- **解决**: 编写自定义 NSIS 脚本 `build/installer.nsh`，使用 `!macro customHeader` 强行锁定注入 `!define APP_INSTALLATION_APP_NAME "OpenScanLink"`。

### 🔴 坑 5: 卸载流程繁琐且出现“全新安装”错别字
- **现象**: 卸载时需要点好几步，且界面上出现了奇怪的“全新安装”提示。
- **原因**: 开发期间频繁测试“为所有用户安装(perMachine)”和“仅为当前用户安装”，导致 Windows 注册表权限残留冲突。
- **解决**: 在 `build/installer.nsh` 中编写 `!macro customUnInit` 宏，拦截标准卸载向导，改为**原生弹窗确认后立即执行 `SetSilent silent` 静默秒删**，实现了极简的“一键卸载”。

### 🔴 坑 6: 双面扫描时，第一页图片边缘内容被异常裁剪缺失
- **现象**: 扫描双面文件时，反面正常，但正面的文字边缘缺失了一块。
- **原因**: `Magick.NET` 中的去黑边方法 `Trim()` 容差 (`ColorFuzz`) 设为了 40%，过于激进，把正常文字的边缘误当成了黑边；且在纠偏 (`Deskew`) 后进行了二次 `Trim()`，加剧了误伤。
- **解决**: 将 `ColorFuzz` 容差降低至 10%，并删除了纠偏后的第二次 `Trim()`，重新编译 C# 引擎后完美解决。

### 🔴 坑 7: 扫描处理极度缓慢，5张双面需要 1分18秒
- **现象**: 硬件进纸速度还可以，但扫描完成后，网页前端迟迟等不到处理完成的图片，甚至有时会出现管道通信截断导致假死。
- **原因**: 
  1. 开启了 NAPS2 内部极其缓慢的“单线程软件纠偏（Software Deskew）”，导致 CPU 阻塞。
  2. 原有逻辑每次扫描后都会强杀并重启 C# 进程，频繁经历 .NET 和 TWAIN 驱动的冷启动。
  3. C# 进程输出的超长 JSON 字符串在 Node.js 管道中被分块（Chunking），导致解析失败。
- **解决**: 
  1. 彻底禁用 NAPS2 内置纠偏，保留极速的“色阶映射去底灰”和瞬间的矩阵翻转，将单张后处理时间压缩至 **1秒内**。
  2. 将 C# 底层改造成**常驻内存的守护进程（Daemon）**，利用持久化的标准输入输出管道通信。
  3. Node.js 端增加 `stdoutBuffer` 缓冲拼接机制，确保完整读取 JSON。目前 5 张试卷总耗时已被压榨至极致的 **30秒左右**。

### 🔴 坑 8: 目标机器未安装 .NET 8 运行时导致无法运行
- **现象**: 网页请求 `/api/device/list` 报错 `You must install .NET to run this application`，错误码 `0x80008083`，提示找不到 `hostfxr.dll`。
- **原因**: 之前为了减小体积，编译参数使用了 `--self-contained false`（依赖框架模式），导致在没有安装 .NET 8 x86 运行时的电脑上直接崩溃。
- **解决**: 将编译命令修改为 `dotnet publish -c Release -r win-x86 --self-contained true -p:PublishSingleFile=true`，即**独立部署模式**。这样会将 .NET 运行时直接打包进 EXE 中，即使目标机器没有任何环境也能直接运行，彻底告别运行时依赖缺失问题。

### 🔴 坑 9: 彩色模式下噪点多、字迹发灰不清晰
- **现象**: 彩色扫描出来的试卷底色不够纯白，且红笔或铅笔批注的字迹比较浅，不如官方驱动软件处理得清晰。
- **原因**: 彩色模式下未进行针对性的对比度拉伸，且激进的白场阈值导致浅色字迹丢失。
- **解决**: 在图像处理管线中引入 `Level(8%, 90%)` 色阶算法，强制将暗部下压 8% 加深黑色字迹，同时配合 `Gamma(0.9)` 和 `Modulate(120%)` 提升整体浓重感和饱和度。最终实现了字黑、红艳、底白的高品质效果。

### 坑 10: 为什么扫描仪启动需要 7~8 秒的延迟？能否像官方软件一样 3 秒启动？

**现象**：
点击“开始扫描”后，物理扫描仪大概需要等待 7~8 秒后才会开始第一张纸的吸入。而使用原厂自带的 AVScan X 软件时，点击后仅需 3 秒左右就会进纸。

**深度分析与原因**：
根据 TWAIN 协议规范，启动扫描仪需要经历严格的状态机跃迁：
- State 1：扫描仪断开连接。
- State 2：加载 TWAIN 驱动 DLL。
- State 3：打开设备物理连接（USB 握手并点亮灯管）。
- State 4：完全准备好并开始进纸。

原厂软件（AVScan X）在用户打开软件时，就会**在后台秘密将扫描仪强制挂载并保持在 State 4 状态（长连接机制）**。因此当你点击扫描时，它只需下发一个进纸指令。
但作为第三方中间件，如果我们也强行霸占 State 4 状态，会导致两个致命后果：
1. **设备被独占锁定**：其他任何软件（如微信、WPS、打印机面板）都将提示“扫描仪正被占用”。
2. **极易引发硬件死锁**：一旦中间件崩溃或刷新页面，而未及时发送 `MSG_CLOSEDSREQ`（关闭数据源指令），扫描仪的灯管会一直亮着，必须拔电源重启硬件（这也是我们早期踩过坑的原因）。

因此，底层 `NAPS2.Core` 引擎采用了**“用完即焚”的安全策略**。每次触发扫描，都必须完整走一遍 State 1 -> State 4 的握手和自检流程。这 7 秒的延迟是 TWAIN 规范保障驱动稳定性和不霸占设备所必须付出的**合理物理开销**，在第三方框架下无法也不应被强行抹除。

### 🔴 坑 11: 生产环境打包后 sharp 模块因中文路径或缺少 DLL 崩溃
- **现象**: `npm run build:win` 打包成 exe 安装后，软件启动时报错 `The specified module could not be found. ...\sharp-win32-x64.node`，导致核心功能瘫痪。
- **原因**: 
  1. 用户将软件安装在了包含中文或特殊字符的路径下，Windows 底层 `LoadLibrary` API 加载非纯英文路径下的 `.node` 扩展时产生乱码并失败。
  2. `electron-builder` 默认将 `sharp` 的一堆外置 `libvips` DLL 压缩在了 `app.asar` 中，导致 `.node` 文件在运行时找不到同目录下的动态链接库。
- **解决**: 
  1. 在 `package.json` 的 `build` 配置中新增 `"asarUnpack": ["**/node_modules/sharp/**/*"]`，强制将 `sharp` 模块从黑盒中物理释出。
  2. 在 NSIS 自定义脚本 `build/installer.nsh` 中，使用 `!macro customInit` 强制将默认安装路径覆盖为纯英文（如 `$PROGRAMFILES\OpenScanLink`），并利用 `System::Call` 和 `WideCharToMultiByte` API 在 `!macro verifyDir` 生命周期钩子中进行强校验，若用户手动选择了中文路径则弹出警告并阻止进入下一步安装。

## 4. 目录结构核心说明
- `/bridge/ScannerService/` : C# 底层核心扫描服务源码，最终编译为独立的可执行文件。
- `/src/main/` : Electron 主进程代码（负责与 C# 通信、开启 HTTP 服务、系统级操作）。
- `/src/renderer/` : Electron 渲染进程代码（客户端 UI）。
- `/web/demo/` : 纯网页端对接测试 Demo (Vue3)。
- `/build/installer.nsh` : NSIS 自定义安装/卸载逻辑脚本。

## 5. 结语
通过本次重构，`OpenScanLink` 已经具备了稳定可靠的运行能力，彻底解决了环境隔离、安装包臃肿、图像处理误伤等核心痛点，并提供了优秀的静默化用户体验。
